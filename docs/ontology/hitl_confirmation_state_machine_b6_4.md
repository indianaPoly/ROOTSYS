# Human-in-the-Loop Confirmation State Machine (B6-4)

This document defines the R3 human-in-the-loop state machine for candidate link confirmation.

## Scope
- Candidate link lifecycle from R2 output to human decision
- State transitions driven by `confirmLink` and `rejectLink`
- Guard conditions, idempotency behavior, and conflict handling

## Goals
- Make review decisions explicit, auditable, and deterministic.
- Prevent ambiguous updates on candidate links.
- Keep R3 outcomes aligned with ontology and action specs.

## Actors
- `operator`: submits confirm/reject decisions
- `reviewer`: validates or overrides according to policy
- `system`: executes transition, emits audit events

## State Set

- `candidate`
  - generated by R2 probabilistic linking
- `in_review`
  - optional temporary lock state while action is being processed
- `confirmed`
  - candidate accepted as valid relation
- `rejected`
  - candidate rejected with reason
- `promoted`
  - candidate superseded by deterministic or confirmed canonical link
- `discarded`
  - invalidated due to data retirement or dedup cleanup

## Transition Table

| From       | Event                    | To         | Guard |
|------------|--------------------------|------------|-------|
| candidate  | start_review             | in_review  | candidate exists and not locked |
| in_review  | confirmLink              | confirmed  | policy allow + payload valid |
| in_review  | rejectLink               | rejected   | policy allow + payload valid |
| candidate  | confirmLink (fast-path)  | confirmed  | policy allow + optimistic lock ok |
| candidate  | rejectLink (fast-path)   | rejected   | policy allow + optimistic lock ok |
| candidate  | deterministic_override   | promoted   | matching R1 confirmed link exists |
| confirmed  | invalidate_confirmed_link| discarded  | governance/admin decision |
| rejected   | reopen_for_review        | in_review  | reviewer/admin only |

## Transition Rules

## confirmLink
- Allowed from: `candidate`, `in_review`
- Required payload:
  - `link_id`
  - `justification`
  - optional `confidence_override`
- Effects:
  - state -> `confirmed`
  - create/update `has_cause` confirmed relation
  - emit audit events: `action.policy_decision`, `action.state_transition_succeeded`

## rejectLink
- Allowed from: `candidate`, `in_review`
- Required payload:
  - `link_id`
  - `reason`
  - optional `reason_code`
- Effects:
  - state -> `rejected`
  - keep candidate history row (no hard delete)
  - emit audit events with rejection reason

## deterministic_override
- Trigger: R1 later confirms same `(from_id, to_id)` pair
- Allowed from: `candidate`
- Effects:
  - state -> `promoted`
  - canonical `has_cause` remains authoritative

## Guards and Validation

- Reject transitions when link not found: `R3_LINK_NOT_FOUND`
- Reject invalid source state: `R3_INVALID_TRANSITION`
- Reject policy deny: `R3_PERMISSION_DENIED`
- Reject stale update version: `R3_CONCURRENT_MODIFICATION`
- Reject malformed payload: `R3_ACTION_VALIDATION_FAILED`

## Concurrency and Idempotency

- Use optimistic lock with `version` or `updated_at_unix_ms` precondition.
- `action_id` is idempotency key for action execution.
- Replayed action with same `action_id` returns prior decision outcome.

## Audit Requirements

Every successful or failed transition writes:
- `event_type`
- `action_id`
- `link_id`
- `from_state`
- `to_state` (or null on failure)
- `subject_id`
- `decision` and `policy_code`
- `reason`/`reason_code` (when applicable)
- timestamp fields

## UI/Workflow Expectations

- Candidate queue sorted by confidence band (`high -> medium -> low`).
- Reviewer must see explain tokens and source lineage before decision.
- Rejected candidate can be reopened only by reviewer/admin.

## Invariants

- A link cannot be both `confirmed` and `rejected` simultaneously.
- R3 never mutates R1 deterministic provenance fields.
- Transition history is append-only in audit trail.
