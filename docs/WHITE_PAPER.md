# WHITE PAPER
이 백서는 제조업 환경에서 빈번히 발생하는, 복잡하고 분산된 불량 원인 분석 문제를 Palantir Foundry의 계층형 아키텍처를 활용하여 어떻게 효과적으로 해결하고 분석 시간을 단축할 수 있는지에 대한 청사진을 제시합니다. 특히, 데이터 통합과 온톨로지 구축을 통해 산재된 정보를 의미 있는 지식으로 전환하고, 체계적인 액션 및 사용자 인터페이스를 통해 신속하고 정확한 의사결정을 지원하는 프레임워크를 설명합니다.

## 1. 시나리오 개요: "불량 원인 분석 시간 단축"

제조업 현장에서는 "불량 원인이 많고, 여러 시스템에 흩어져 있어 분석에 오랜 시간이 소요된다"는 문제가 고질적으로 발생합니다. 이러한 현실적인 어려움은 품질 관리 및 생산 효율성 저하의 주요 원인이 됩니다. 본 백서는 이러한 복잡한 불량 원인들을 담당자가 OS(Operational System)를 활용하여 한 화면에서 종합적으로 분석하고, 이를 통해 전체 분석 시간을 획기적으로 단축하는 시나리오를 Palantir Foundry의 각 레이어에 1:1로 매칭하여 설명합니다.

## 2. Palantir Foundry 계층 구조 및 역할

Palantir Foundry는 다양한 데이터를 통합하고, 실제 세상의 엔티티와 관계를 모델링하며, 사용자에게 실행 가능한 통찰력을 제공하는 운영 시스템입니다. Foundry의 핵심은 온톨로지이며, 이는 조직의 디지털 자산을 현실 세계의 객체(예: 공장, 장비, 제품) 및 개념(예: 고객 주문, 금융 거래)과 연결하는 운영 계층 역할을 합니다. 이러한 Foundry의 아키텍처는 다음과 같은 주요 레이어로 구성됩니다.

### 2.1 External System

불량 원인에 대한 원천 데이터가 존재하는 다양한 외부 시스템은 Foundry 아키텍처의 가장 하단에 위치합니다. 이 레이어는 "원인 후보 데이터가 실제로 존재하는 곳"으로서, MES(생산 실행 시스템), QMS(품질 관리 시스템), EAM(자산 관리 시스템), ERP(전사적 자원 관리), 엑셀 파일, 이미지 파일 등 다양한 형태로 데이터를 보유합니다.

- 현실 문제: 불량 원인 데이터가 MES의 생산 기록, QMS의 품질 클레임 기록, 엑셀 파일의 수동 점검표, 이미지 파일의 불량 사진 등 여러 시스템에 파편화되어 흩어져 있습니다. 각 시스템은 고유한 데이터 형식과 저장 방식을 사용합니다.
- 역할: 이 레이어는 "원인이 많다"는 사실 자체를 만들어내는 원천 소스입니다. 즉, 불량 발생의 다양한 측면을 기록하는 시스템들이며, 이들의 이질적인 데이터가 통합 및 분석의 출발점이 됩니다.

### 2.2 Data Integration Layer

데이터 통합 레이어는 다양한 외부 시스템에 흩어진 이질적인 데이터를 Foundry 플랫폼 내에서 통일된 형식으로 수집, 표준화 및 정규화하는 역할을 담당합니다. 이 과정은 ETL(Extract, Transform, Load) 파이프라인을 통해 이루어지며, Foundry는 일반적인 ETL/ELT 솔루션을 넘어서는 고도로 구성 가능한 도구 세트를 제공합니다.

- 현실 문제: 각 외부 시스템의 데이터는 형식, 구조, 저장 방식이 달라, 사람이 직접 확인하고 통합해야 하는 비효율성이 발생합니다. 이는 "흩어져 있어서 보기 힘들다"는 문제를 야기합니다.
- 역할: 이 레이어는 "흩어져 있어서 보기 힘들다"는 문제를 해결합니다.
    - 공통 계약(Common Contract) 정의: 외부 시스템을 직접 통합하기보다는, OS(Operational System)가 필요로 하는 최소한의 공통 형식(표준 이벤트, 표준 레코드)을 먼저 정의합니다. 예를 들어, Defect Record, Cause Candidate Record, Evidence Record, Link Record와 같은 표준 데이터 형식을 설정하며, 이러한 정의는 Foundry Ontology의 Object 및 Link Types에 해당합니다.
    - 파이프라인 구축: Foundry는 확장 가능한 데이터 연결 프레임워크를 통해 구조화, 비구조화, 반구조화된 모든 유형의 소스 시스템에 연결하며, 배치, 마이크로 배치, 스트리밍과 같은 모든 주요 데이터 전송 방식을 지원합니다. 데이터 변환을 위해 다양한 컴퓨팅 모드를 활용하여 결과물 데이터셋을 생성하는 확장 가능한 데이터 빌드 시스템을 제공합니다. Pipeline Builder 애플리케이션은 강력하고 안전하며 빠르고 유연한 데이터 파이프라인 전달을 가능하게 합니다.
        - Source (소스): 데이터가 어디서 발생하는지 명시합니다 (예: MES DB, EAM API, S3 버킷 내 Excel/CSV 파일).
        - Extract (추출): 소스에서 데이터를 어떻게 가져올지 정의합니다 (예: SQL 쿼리, API 호출, 특정 디렉토리 스캔, CDC(Change Data Capture) 방식, 배치 스케줄). Foundry는 데이터의 원본을 보존하고 접근 가능하도록 Data Lineage 기록을 유지합니다.
        - Transform (변환): 추출된 원본 데이터를 온톨로지의 표준 객체 형식에 맞춰 구조화하고 정규화합니다. 이 과정에서 데이터 타입 변환, 단위 통일, 코드 표준화, 결측치 처리 등이 이루어집니다. AIP의 AI 기반 파싱 기능은 사전 훈련되거나 맞춤화된 AI 모델(예: NLP 모델)을 파이프라인 내에 통합하여 개체명 인식, 요약과 같은 정교한 작업을 수행할 수 있습니다. 모든 데이터는 아파치 파케이(Apache Parquet), 아브로(Avro)와 같은 표준 형식으로 저장되며, 대규모 배치 처리를 위한 아파치 스파크, 실시간 스트림 처리를 위한 아파치 플링크와 같은 오픈소스 런타임을 사용합니다.
        - Load (적재): 변환된 데이터를 Foundry의 Object Store에 특정 Object Type 및 Link Type으로 적재합니다. 데이터의 멱등성(Idempotency)을 고려하여 중복 데이터를 효율적으로 처리해야 합니다. 실패 처리 메커니즘을 통해 파싱 실패, 필수 필드 누락 등은 DLQ(Dead Letter Queue)에 저장하여 데이터 유실을 방지합니다.
        - Keying (키 전략): 서로 다른 소스의 데이터를 하나의 온톨로지 객체 또는 관계와 연결하기 위한 고유 식별자 및 매칭 규칙을 정의합니다. 이는 원천 ID 매칭, 공통 조인 키 활용, 또는 시간 및 속성 기반의 확률적 매칭을 포함할 수 있습니다.
        - Quality & Lineage (품질 및 계보): 데이터 품질을 보장하기 위한 유효성 검사 규칙, 실패 처리 로직, 그리고 데이터의 원본 소스부터 최종 온톨로지 객체까지의 전체 흐름(Lineage)을 기록하는 메커니즘을 포함합니다. Foundry의 데이터 리니지 기능은 데이터가 어디서 와서 어떤 과정을 거쳐 어떻게 활용되는지를 시각적으로 보여줍니다.
    - 파이프라인 일정 관리: 생산 파이프라인의 일정을 효과적으로 관리하기 위한 지침이 제공됩니다. 대부분의 경우 파이프라인 당 하나의 일정만 가지는 것이 권장되며, 데이터셋이 두 개의 다른 일정에 의해 빌드되면 큐잉과 지연이 발생할 수 있습니다. 또한, 재시도 메커니즘을 활용하여 일시적인 오류로 인한 빌드 실패를 자동으로 처리할 수 있습니다.

### 2.3 Ontology Layer (온톨로지 레이어)

온톨로지 레이어는 통합된 데이터를 현실 세계의 의미 있는 객체(Object), 속성(Property), 관계(Link)로 모델링하여 조직의 개념적 이해를 구축하는 Palantir Foundry의 핵심 요소입니다. 이는 데이터 사일로를 해소하고, 데이터와 AI, 사람 간의 협업을 가능하게 하는 공유 언어 역할을 합니다. Foundry의 온톨로지는 오브젝트 유형, 속성, 링크 유형, 액션 유형에 매핑함으로써 조직의 세계에 대한 완전한 그림을 만듭니다.

- 현실 문제: 데이터 통합을 통해 모든 정보가 한곳에 모여도, "이것이 같은 불량에 대한 원인들"이라는 의미론적 구조가 부족하면 여전히 분석에 어려움이 있습니다. 즉, 단순히 데이터가 모이는 것을 넘어 "데이터를 '원인'이라는 의미 단위로 묶는" 과정이 필요합니다.
- 역할: 이 레이어는 "많은 데이터 → 많은 원인 객체"로 변환하는 역할을 수행합니다.
    - 객체(Objects) 정의: Defect, Cause, CompositeCause, Evidence와 같은 객체 유형을 정의하여 실제 세계의 엔티티를 표현합니다. 예를 들어, MES 기록에서 추출된 정보를 "설비 마모"라는 Cause 객체로, QMS 클레임을 "작업자 숙련도 부족"이라는 Cause 객체로, 환경 로그를 "습도"라는 Cause 객체로 변환합니다.
    - 속성(Properties) 정의: 각 객체가 가질 수 있는 특징(예: Cause 객체의 cause_type, description, confidence)을 정의합니다.
    - 관계(Links) 정의: 링크 유형은 두 객체 유형 사이의 관계를 정의합니다. 예를 들어, Defect has_cause Cause, Cause supported_by Evidence, Cause combines_to CompositeCause와 같은 관계를 정의하며, 이는 객체들 간의 단일 인스턴스 관계를 가리킵니다. Foundry SDK를 사용하여 개발자는 온톨로지의 객체와 관계를 마치 일반적인 Python 클래스와 메서드처럼 다룰 수 있습니다.
    - 관계 생성 규칙 (R1, R2, R3): Foundry OS에서 Defect-Cause-Evidence-CompositeCause 관계를 생성하는 파이프라인은 단순히 데이터를 옮기는 것이 아니라, 객체 간의 관계를 생성하는 규칙을 실행합니다.
        - R1 (결정론적 링크 - Deterministic Links): 데이터 내에 lot_id, defect_id, equipment_id와 같은 명확하고 신뢰성 높은 키가 존재하여 객체 간 관계를 100% 확실하게 자동 생성할 수 있는 경우에 적용됩니다. 예를 들어, MES의 특정 defect_id와 QMS의 동일한 defect_id를 가진 클레임을 직접 연결하는 경우입니다.
        - R2 (확률적 링크 - Probabilistic Links): 키가 모호할 때는 "후보"로만 연결하며, 시간적 근접성(±30분), 동일 라인/설비, 유사 불량 코드 패턴 등 통계적 상관관계를 통해 객체 간에 관계가 있을 가능성이 높은 경우에 적용됩니다. 파이프라인은 이때 confidence score와 함께 candidate_of와 같은 후보 링크를 생성하며, 이는 아직 확정되지 않았음을 나타냅니다. 확률적 레코드 연결에 대한 집계 쿼리 분석은 불완전성이나 데이터의 모호성으로 인해 불확실성이 존재할 수 있음을 보여줍니다.
        - R3 (인간 개입 링크 - Human-in-the-Loop Links): R2에서 생성된 확률적 후보 링크를 현업 전문가가 검토하고, 특정 Action을 통해 수동으로 확정하거나 반려하는 규칙입니다. 이 과정에서 사용자의 결정 및 근거(Evidence)는 Action 레이어를 통해 감사 가능한 기록으로 남습니다.

### 2.4 Action / Policy / Audit Layer (액션 / 정책 / 감사 레이어)

액션/정책/감사 레이어는 Foundry 온톨로지에서 정의된 객체 및 관계에 대한 모든 변경 및 의사결정 과정을 시스템화하고, 이를 감사 가능하도록 기록하는 역할을 합니다. 액션 유형은 사용자가 한 번에 객체, 속성 값, 링크에 대해 수행할 수 있는 변경 또는 편집의 집합에 대한 스키마 정의입니다.

- 현실 문제: 불량 원인 분석 과정에서 "이 원인은 유효하다"와 같은 중요한 의사결정이 사람의 말이나 문서로만 남아, 누가, 언제, 어떤 근거로 결정했는지 불명확한 경우가 많습니다. 이는 추적성(traceability)과 거버넌스(governance)의 부족으로 이어집니다.
- 역할: 이 레이어는 "분석 행위 자체를 자동화하고 기록하는 레이어"입니다.
    - 액션(Actions) 정의: "이 원인은 유효하다"(approveCause), "이 두 가지가 복합 원인이다"(confirmCompositeCause)와 같이 온톨로지 객체의 상태를 변경하거나 관계를 생성하는 비즈니스 프로세스를 Action으로 정의합니다. Action은 제출 기준, 권한 및 특정 온톨로지 객체 또는 링크에 대한 수정 유형을 정의하는 정책을 포함할 수 있습니다. Foundry SDK는 온톨로지에 정의된 액션(예: update_shipment_status)을 마치 일반적인 메서드처럼 호출할 수 있게 하여 개발 속도와 데이터 조작의 안전성을 높입니다.
    - 정책(Policies) 시행: 모든 변경이 검증되고, 권한이 부여되며, 사전에 정의된 비즈니스 규칙과 일치하도록 보장하는 정책이 적용됩니다.
    - 감사(Audit) 기록: 모든 Action은 누가, 언제, 어떤 유형의 Action을 수행했으며, 어떤 객체와 속성을 변경했는지에 대한 상세한 감사 로그(audit log)를 생성합니다. 이 불변의 기록은 모든 활동에 대한 완전하고 검색 가능한 이력을 제공하여, 추적성, 보안 및 규정 준수를 가능하게 합니다. Foundry는 데이터 접근 권한을 세밀하게 제어하고 모든 사용자 행동을 로깅하여 감사 추적을 가능하게 합니다.

### 2.5 User / App Layer

사용자/앱 레이어는 최종 사용자가 Foundry 온톨로지의 데이터를 탐색하고, 분석하고, Action을 수행할 수 있도록 하는 직관적인 인터페이스와 애플리케이션을 제공합니다. 이는 복잡한 데이터와 모델을 비기술적인 사용자도 쉽게 활용할 수 있도록 돕습니다.

- 현실 문제: 담당자가 불량 원인 분석을 위해 여러 시스템(MES, QMS, 엑셀, 파일)을 오가며 데이터를 확인해야 하는 비효율성이 발생하며, 이는 분석 시간을 지연시킵니다.
- 역할: 이 레이어는 "담당자가 한 화면에서 원인을 보는" 환경을 제공하여 "담당자가 시간을 아끼는 지점"이 됩니다.
    - 통합된 사용자 인터페이스(UI): CAPA App과 같은 웹 UI를 통해 Defect에 연결된 모든 Cause 목록, CompositeCause 구조, Evidence (사진, 로그), 그리고 Action 버튼 등을 한 화면에서 볼 수 있도록 설계됩니다.
    - 분석 도구 통합: Foundry의 Quiver(시계열 데이터 분석), Workshop(맞춤형 애플리케이션 구축), Object Explorer(객체 탐색)등 다양한 분석 및 운영 도구를 활용하여, 사용자가 특별한 전문 지식 없이도 데이터를 탐색하고 심층 분석할 수 있게 합니다.
    - AI 솔루션 활용: AI 기반 솔루션을 활용하여 사용 편의성을 높이고 분석 시간을 단축하며, 불량 원인 파악 및 관리를 개선합니다. LLM(Large Language Model)과 SMC-S(Sequential Monte Carlo Samplers)를 결합한 온라인 추론 방법은 자연어 가설을 활용하여 규칙 유도 작업에서 기준 모델과 인간의 성능을 능가하는 것으로 나타났습니다. 이러한 모델은 퍼지(fuzzy)하고 확률적인 규칙을 추론하는 인간의 경향을 더 잘 반영하며, 정보 획득(information gain)을 통한 능동 학습(active learning)을 활용하여 경쟁 가설을 최적으로 해결합니다.

## 3. Layer별 시나리오 매칭 및 해결 과정

위에 설명된 Palantir Foundry의 각 레이어는 "불량 원인이 많아서 분석에 오래 걸린다"는 시나리오의 각 부분을 명확히 해결합니다.

| 시나리오 요소 | 담당 레이어 | 역할 및 문제 해결 |
| --- | --- | --- |
| **원인이 많다** | External Systems | MES, QMS, EAM, 엑셀, 파일 등에서 불량 원인과 관련된 수많은 원천 데이터를 생성하고 저장합니다. 이 레이어는 불량 발생의 근본적인 "사실"을 만들어냅니다. |
| **흩어져 있다** | Data Integration Layer | 다양한 외부 시스템에 흩어진 이질적인 형식의 데이터를 공통 계약(표준 형식)에 맞춰 수집, 변환, 정규화하고 Foundry 내부로 적재합니다. 이를 통해 분산된 데이터를 하나의 통일된 뷰로 모읍니다. |
| **원인으로 묶는다** | Ontology Layer | 통합된 데이터로부터 `Defect`, `Cause`, `Evidence`, `CompositeCause`와 같은 의미 있는 객체를 생성하고, 이들 간의 관계(`has_cause`, `supported_by`, `combines_to`)를 정의하여 데이터를 "원인"이라는 개념적 단위로 묶습니다. 이를 통해 개별 데이터 조각들이 특정 불량의 원인으로서 어떻게 연결되는지 의미론적으로 이해할 수 있게 됩니다. |
| **분석 판단을 기록한다** | Action / Policy / Audit Layer | 불량 원인 분석 과정에서 발생하는 "원인 확정", "후보 반려" 등 모든 의사결정을 `Action`으로 정의하고 실행하며, 이 모든 `Action`에 대한 상세한 감사 로그를 자동으로 기록합니다. 이는 누가, 언제, 무엇을, 어떤 근거로 결정했는지에 대한 명확한 추적성과 거버넌스를 보장합니다. |
| **한 화면에서 본다** | User / App Layer | `CAPA App`과 같은 사용자 인터페이스를 통해 담당자가 모든 `Defect`, 관련된 `Cause` 후보 및 확정된 `Cause`, `Evidence` (사진, 문서 등)를 한 화면에서 보고 상호작용할 수 있도록 합니다. 직관적인 UI/UX는 여러 시스템을 오갈 필요 없이 신속한 분석과 의사결정을 가능하게 하여 "담당자가 시간을 아끼는 지점"을 제공합니다. |

## 4. 파이프라인 정의의 중요성: 관계 생성의 공장

Palantir Foundry에서 온톨로지 개념이 진정한 가치를 발휘하기 위해서는, 온톨로지에 정의된 객체들(Defect, Cause, Evidence 등) 간의 관계(Link)를 지속적이고 자동적으로 생성하며 관리하는 파이프라인이 필수적입니다. 파이프라인은 단순히 데이터를 이동시키는 것을 넘어, 외부 시스템의 원천 데이터로부터 의미 있는 관계를 도출하고 온톨로지에 반영하는 "관계 생성의 공장" 역할을 수행합니다.

### 4.1 파이프라인 정의의 6가지 핵심 요소

각 외부 시스템별 파이프라인을 정의할 때는 다음 6가지 요소를 명확히 해야 합니다.

- Source (소스): 데이터가 어디서 발생하는지 명시합니다. Foundry는 다양한 데이터 소스에 연결하는 작업을 지원하며, 일괄처리, 스트리밍, 미디어, CDC 동기화 및 가상 테이블을 활용할 수 있습니다.
- Extract (추출): 소스에서 데이터를 어떻게 가져올지 정의합니다. 이는 쿼리, CDC 또는 배치 주기 등 다양한 방식으로 이루어질 수 있습니다.
- Keying (키 전략): 서로 다른 소스의 데이터를 하나의 온톨로지 객체 또는 관계와 연결하기 위한 고유 식별자 및 매칭 규칙을 정의합니다. 키 설계는 통합의 80%를 차지하는 핵심 단계입니다.
    - 레벨 1: 원천 ID가 있는 경우: MES의 defect_id, QMS의 claim_id 등과 같이 원천 시스템에서 제공하는 고유 ID를 그대로 source_key로 저장합니다.
    - 레벨 2: 공통 조인 키가 있는 경우: lot_id + product + occurred_at window와 같이 여러 필드의 조합을 통해 데이터를 조인할 수 있는 경우입니다.
    - 레벨 3: 키가 없는 경우: 현실에서 흔히 발생하는 경우로, 시간/라인/제품/작업조/설비 조합으로 확률 매칭을 시도하거나, 사람이 최초에 수동으로 연결(링크)하는 방식으로 시작할 수 있습니다. MVP(Minimum Viable Product) 단계에서는 레벨 1~2만 자동으로 처리하고, 레벨 3은 수동 링크로 시작하는 것이 가장 현실적입니다.
- Transform (변환): 추출된 원본 데이터를 온톨로지의 표준 객체 형식에 맞춰 구조화하고 정규화합니다. 이 과정에서 시간대/포맷 통일(UTC vs KST), 코드 표준화(불량코드, 설비코드), 단위 통일(mm, cm, μm), 결측치 처리, 중복 제거 등의 규칙을 정의하고 적용합니다. 이 매핑표가 곧 “통합 설계서”가 됩니다.
- Load (적재): 변환된 데이터를 Foundry의 Object Store에 특정 Object Type 및 Link Type으로 적재합니다. 적재 시 Idempotency(멱등성)를 보장하여 동일한 데이터가 두 번 들어와도 결과가 한 번과 같도록 key + upsert 전략을 필수적으로 사용해야 합니다.
- Quality & Lineage (품질 및 계보): 데이터 품질을 보장하기 위한 유효성 검사 규칙(예: 필수 필드 누락 여부), 실패 처리 로직(예: 파싱 실패 행은 DLQ(Dead Letter Queue)에 저장), 그리고 데이터의 원본 소스부터 최종 온톨로지 객체까지의 전체 흐름(Lineage)을 기록하는 메커니즘을 포함합니다.

### 4.2 관계 생성 규칙: R1, R2, R3의 활용

파이프라인은 단순히 데이터를 옮기는 것이 아니라, 객체 간의 관계를 생성하는 규칙(Rules)을 실행합니다. 이 규칙은 세 가지 계층으로 나뉘어 복잡한 제조 환경의 특성을 반영하며, 이는 유한한 자원과 불확실한 데이터를 처리하기 위한 현실적인 접근 방식입니다.

- R1 (결정론적 링크 - Deterministic Links)
    - 설명: 데이터 내에 lot_id, defect_id, equipment_id와 같이 명확하고 신뢰성 높은 키가 존재하여 객체 간 관계를 100% 확실하게 자동 생성할 수 있는 경우에 적용됩니다. 이는 같은 lot_id, 같은 equipment_id, 같은 defect_id, 시간 범위가 확실히 동일과 같은 명확한 조건에 기반합니다.
    - 알고리즘 선택 이유: Foundry의 Object ID 및 Link ID 관리가 일관된 데이터 모델링을 보장하고 데이터의 무결성을 유지하는 데 필수적입니다. 데이터의 불변성과 추적성을 보장하는 가장 확실한 방법으로, 데이터 소스의 고유 식별자를 활용하여 정확한 관계를 생성합니다. 이는 데이터 리니지 확보의 기본이 됩니다.
    - 구현 방법: 파이프라인에서 추출된 MES defect_id를 Defect 객체의 고유 ID로 사용하고, 동일한 lot_id를 가진 검사 결과를 Defect 객체에 연결하는 방식 등이 해당됩니다. 이는 client.ontology.objects.Shipment.where(status="Delayed").all()과 같은 Foundry SDK를 통한 객체 조회 및 연결된 객체 탐색 기능으로 구현될 수 있습니다.
- R2 (확률적 링크 - Probabilistic Links)
    - 설명: 키가 모호할 때는 "후보"로만 연결하며, 시간적 근접성(±30분), 동일 라인/설비, 유사 불량 코드 패턴, 동일 작업조 등 명확하지 않은 조건에서 통계적 또는 경험적 가설에 기반하여 객체 간에 관계가 있을 가능성(confidence score)을 부여하고 후보 링크를 생성합니다.
    - 알고리즘 선택 이유: 제조업 데이터는 불완전하거나 모호한 경우가 많으므로, 100% 결정론적인 연결만이 가능한 것은 아닙니다. Probabilistic Record Linkage 방법은 두 레코드가 동일한 실제 엔티티를 나타낼 확률을 계산하여 불확실성을 관리합니다. LLM(Large Language Models)과 같은 고급 AI 모델을 활용하여 자연어 가설을 기반으로 확률적 추론을 수행함으로써, 명확하지 않은 규칙을 유도하고 인간의 추론 방식에 더 가깝게 모델링할 수 있습니다. 특히, Sequential Monte Carlo Samplers (SMC-S)는 새로운 데이터가 수집됨에 따라 자연어 가설(particle) 풀을 추적하고 업데이트하여 근사적인 추론과 예측을 제공합니다.
    - 구현 방법: 파이프라인에서 시간 범위나 특정 속성 값(예: temperature_sensor_value와 Cause 간의 임계치 초과 여부)을 기반으로 Cause 객체를 생성하고, confidence 속성을 부여한 뒤 Defect 객체와 candidate_of 유형의 링크로 연결합니다. LLM 기반 모델은 LLM proposal distribution을 사용하여 신념 상태(belief state)를 수정하고, information gain을 최대화하는 실험을 제안하여 경쟁 가설을 최적으로 해결합니다.
- R3 (인간 개입 링크 - Human-in-the-Loop Links)
    - 설명: R2에서 생성된 확률적 후보 링크를 현업 전문가가 검토하고, 특정 Action(approveLink, rejectLink)을 통해 수동으로 확정하거나 반려하는 규칙입니다. 이 과정에서 사용자의 결정 및 근거(Evidence)는 Action 레이어를 통해 감사 가능한 기록으로 남습니다.
    - 알고리즘 선택 이유: AI 모델이나 자동화된 규칙만으로는 모든 불확실성을 해소할 수 없으며, 특히 민감하거나 복잡한 의사결정에는 인간의 전문 지식과 판단이 필수적입니다. Human-in-the-loop 접근 방식은 시스템의 정확성과 신뢰성을 높이고, 전문가의 지식을 시스템에 피드백하여 모델을 지속적으로 개선하는 데 기여합니다. 또한, 모든 인간의 개입이 Action으로 기록되므로, 투명성과 감사 가능성을 확보할 수 있습니다.
    - 구현 방법: User/App Layer의 CAPA App 내에서 후보 링크 목록을 제공하고, 사용자가 Action 버튼(예: Approve Cause, Reject Cause, Add Evidence)을 클릭하여 링크를 확정하거나 반려하도록 합니다. 이 Action은 Action/Policy/Audit Layer를 통해 처리되며, 실행된 Action의 모든 정보는 감사 로그에 기록됩니다.

### 4.3 Action 프로토콜: 관계 생성 행위의 시스템화

OS에서 관계(링크) 생성 규칙을 적용하는 것은 단순히 데이터를 연결하는 것을 넘어, "링크 확정", "반려", "근거 첨부"와 같은 의사결정 행위 자체를 Action으로 정의하고 관리하는 것입니다. 이는 Foundry의 Action 기능을 통해 구현되며, 액션 유형은 온톨로지에서 설정된 후 최종 사용자가 객체를 변경하는 데 사용될 수 있습니다.

- Action 이름 및 페이로드 설계:
    - confirmLink: link_id (확정할 링크), justification (확정 사유).
    - rejectLink: link_id (반려할 링크), reason (반려 사유).
    - addEvidenceToLink: link_id, evidence_id (첨부할 근거 객체 ID), description.
- 프로토콜 적용: 이 Action들은 Foundry Branching을 통한 검토 과정을 거치거나, 특정 권한을 가진 사용자만이 실행할 수 있도록 정책이 적용됩니다. 모든 Action의 실행은 누가, 언제, 무엇을 했는지에 대한 Audit Log로 기록되어 투명성과 추적성을 확보합니다. 개발자는 Foundry SDK를 통해 Action을 호출하여 시스템에 정의된 거버넌스와 보안 모델을 통과하는 감사 가능한 트랜잭션을 실행할 수 있습니다.

## 5. 핵심 요약 및 시사점

Palantir Foundry 기반의 OS 아키텍처는 제조업의 고질적인 문제인 복잡하고 분산된 불량 원인 분석을 체계적으로 해결할 수 있는 강력한 프레임워크를 제공합니다. 외부 시스템의 파편화된 데이터를 통합하고, 온톨로지를 통해 의미 있는 객체 및 관계로 구조화하며, 액션 및 정책 레이어를 통해 모든 분석 및 의사결정을 시스템화하고 감사 가능하게 만듭니다. 최종적으로 사용자/앱 레이어는 이 모든 정보를 통합된 환경에서 제공함으로써, 현장 담당자가 신속하고 정확하게 불량 원인을 파악하고 대응할 수 있도록 지원합니다.

이러한 접근 방식은 단순히 데이터를 모으는 것을 넘어, 데이터에 의미를 부여하고, 그 의미를 기반으로 운영을 자동화하며, 모든 과정을 투명하게 관리하는 "디지털 트윈"으로서의 OS를 구현하는 것입니다. 결과적으로 "불량 원인이 많아도 OS를 사용하면 한 번에 분석할 수 있다"는 목표를 달성하여, 제조업의 품질 관리 및 생산 효율성을 획기적으로 개선할 수 있습니다.
